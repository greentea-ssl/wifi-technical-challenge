## 1. CAN ID 割り当て（11bit）

CAN ID は、値が小さいほど優先度が高くなる。

| Bit    | 10-6           | 5-3                  | 2-0                     |
| :----- | :------------- | :------------------- | :---------------------- |
| 用途   | **メッセージ種別** | **送信元デバイス種別** | **送信元デバイスID (サブID)** |

* **メッセージ種別 (Bit 10-6)**:
    * `00000`: **HID状態通知**
    * `00001`: **FWバージョン返答**
    * `00010`: **HID直接コマンド** (HID → 電源基板)
    * `00011`: **HID直接応答** (電源基板 → HID)
    * `00100`: **メイン基板パラメータ設定**
    * `00101`: **メイン基板パラメータ読み出し**
    * `00110`～`00111`: 予約済み
    * `01000`: **電源コマンド**（メイン基板からの指令, 低速バス）
    * `01001`: **電源フィードバック**(電源基板からのフィードバック, 低速バス)
    * `01010`: **電源パラメータ操作要求** (メイン基板 → 電源基板)
    * `01011`: **電源パラメータ応答** (電源基板 → メイン基板)
    * `01100`: **OTA進捗** (HID 中継表示用) — OTA 中の基板が進捗を送り、HID が Wio 表示器へ中継する (v2.1.0 新設)
    * `01101`～`11101`: 予約済み
    * `11110`: **bridge 透過 (応答/診断)** — HID bridge が **常時** PC へ転送する応答・診断用クラス (v2.1.0 で新設)。サブID は通常どおり (severity 例外なし)。詳細は [hid_bridge.md](./hid_bridge.md)
    * `11111`: **テレメトリ** (HID 経由で PC へ転送される。サブID = severity / 詳細は [hid_bridge.md](./hid_bridge.md))
* **送信元デバイス種別 (Bit 5-3)**:
    * `000`: メイン基板
    * `001`: HID
    * `010`: WiFi
    * `011`: 電源基板 (Power Board)
    * `100`～`111`: 予約済み
* **送信元デバイスID (サブID) (Bit 2-0)**:
    * 電源基板、HIDの場合: `000` (単一デバイスのため)
    * `001`～`111`: 予約済み
    * **メッセージ種別 = `11111` (テレメトリ) の場合のみ例外**: サブID は送信元デバイス ID ではなく **severity** (`0` = FATAL ～ `5` = TRACE, `6`-`7` = 予約) として使用。severity 名称と運用ルールの正典は [hid_bridge.md](./hid_bridge.md) を参照。

### bridge 透過クラス (メッセージ種別 `11110`) — v2.1.0 新設

PC が HID の汎用 CAN ブリッジ ([hid_bridge.md](./hid_bridge.md)) 経由で送ったコマンドへの**応答**等を、PC 側で観測するためのクラス。

* HID bridge は **メッセージ種別 `11110` のフレームを常時 PC へ転送する** (ログレベル非依存)。種別 `11111` (テレメトリ) が severity でフィルタされるのに対し、`11110` は常に転送される。
* **送信元デバイス種別 (Bit 5-3) / サブID (Bit 2-0) は通常エンコード** (severity 例外なし)。応答口ごとに `11110 | 送信元 | サブID` で一意の応答 ID を割り当てられる。
* **応答側の制約**: bridge 経由で PC が観測したい応答は、**種別 `11110` の CAN ID で返す**こと。bridge は中身を解釈しない (土管) ため、どの要求にどの ID で応答するかは各「要求口」の仕様で **「要求 ID → 応答 ID (`11110` 系)」** として定義する。
* **識別子**: PC は、要求口に対応する応答 ID (仕様で定義) ＋ 応答ペイロードによって、自身の要求と応答を突合する。
* 既存の応答 (`0x0D8` HID直接応答 / `0x2D8` 電源パラメータ応答 / `0x040` FW返答) は `11110` ではないため bridge の自動転送対象外 (互換維持)。新規に PC 観測したい応答口を設ける場合は `11110` 系で設計する。

> **v2.0.0 での既存ID例外の排除:**
> v1.x までは `0x201` / `0x259` (サブIDを連番として転用) と `0x210` / `0x218` (ビット割当規則完全に外れた既存ID) という 2 種類の例外があったが、v2.0.0 ですべて廃止し上記の rule-compliant な ID に再割当した。サブID = severity となるテレメトリ (`11111`) のみが正式な仕様上の例外として残る。新旧 ID 対応は §「v1→v2 ID 移行表」を参照。

> **v1→v2 ID 移行表:**
>
> | v1.x | v2.0.0 | 用途 |
> |---|---|---|
> | `0x201` | **`0x280`** | パラメータ設定/読み取り要求 |
> | `0x259` | **`0x2D8`** | パラメータ応答 (multi-frame) |
> | `0x210` | **`0x088`** | HID直接コマンド |
> | `0x218` | **`0x0D8`** | HID直接応答 |
>
> 旧 ID は v2.0.0 で**完全に廃止**。両ファーム (HID, PowerBoard, メイン基板) を新 ID へ同時に切り替える必要がある。

### OTA進捗フレーム (メッセージ種別 `01100`) — v2.1.0 新設

OTA 更新中の基板が進捗を送信し、HID が Wio (表示器) へ中継して進捗バーを表示するための診断フレーム。制御には用いない (PowerBoard #4)。

* 送信元は OTA 中の基板 (現状は電源基板 = `0x318`)。**HID 自身の OTA 進捗は I2C 直送のため本フレームは使わない。**
* HID は本種別を受けたら Wio へ I2C (`CMD_OTA_PROGRESS`) で中継する。bridge による PC 転送対象外。
* **payload (DLC 2)**: `Byte 0` = phase (`0`=接続中 / `1`=DL中 / `2`=適用中 / `3`=完了 / `4`=失敗)、`Byte 1` = percent (`0`-`100`、phase=DL中のとき有効)。
* OTA 中は通常制御フレームが停止するため、本フレームの低優先度 (大きい CAN ID) は問題にならない。

## 2. 低速バス
### 2.1. HIDからメイン基板への指令

* **HID → メイン基板：HID状態通知**
    * **CAN ID**: `0x008`
    * **DLC**: 2 バイト
    * **データ**:
        * `Byte 0`: **動作モード (0: ノーマル, 1: マニュアル, 2: デバッグ)**
        * `Byte 1`: **ロボットID**

### 2.2. メイン基板からHIDへの返答
メイン基板はHIDからのHID状態通知指令（0x008）を受信すると、以下のフィードバックを返送する。
* **メイン基板 → HID：HIDに表示したいデータ等**
    * **CAN ID**: `0x040` (`0b 00001 000 000`)
    * **DLC**: 5バイト
    * **データ**:
        * `Byte 0-3`: **ファームウェアバージョン** (ビットごとにyyyyyyym mmmddddd ssssssss ssssssss, sはシリアル値)
        * `Byte 4`: **動作モード (0: ノーマル, 1: マニュアル, 2: デバッグ)** 


### 2.3. メイン基板から電源基板への指令

* **メイン基板 → 電源基板：電源に関する指令値**
    * **CAN ID**: `0x200` (`0b 01000 000 000`)
    * **DLC**: 1バイト
    * **データ**:
        * `Byte 0`: **電源制御指令** (8bit ビットフラグ)
            * `bit0`: **アクチュエータ電源ON** (`0`: アクチュエータ電源停止, `1`: アクチュエータ電源有効)
            * `bit1`: **制御電源OFFフラグ** (このビットが`1`になったら制御電源OFF)
            * `bit7-2`: 予約

### 2.4. 電源基板からメイン基板へのフィードバック

電源基板はメイン基板からの電源制御指令（0x200）を受信すると、以下のフィードバックを返送する。

* **電源基板 → メイン基板：電源フィードバック**
    * **CAN ID**: `0x258`
    * **DLC**: 8 バイト
    * **データ**:
        * `Byte 0`: **停止理由** (8bit ビットフラグ)
            * `bit0`: **メイン基板指令** (メイン基板からの全体停止指令による停止)
            * `bit1`: **HID非常停止** (HID経由の非常停止信号による停止)
            * `bit2`: **遠隔非常停止** (電源基板側が受信した命令による停止)
            * `bit3`: **過電流検出** (設定閾値以上が設定時間継続)
            * `bit4`: **低電圧検出** (セル電圧が設定閾値未満)
            * `bit5`: **LOCAL_ESTOP** (シリアルシェルからの停止)
            * `bit6`: **タイムアウト** (電源制御指令のタイムアウト)(未実装)
            * `bit7`: 予約
        * `Byte 1-2`: **電流検出値** (符号付き16bit整数, リトルエンディアン, mA単位)
        * `Byte 3`: **セル0電圧** (符号なし8bit整数, 0.02V単位, mV÷20)
        * `Byte 4`: **セル1電圧** (符号なし8bit整数, 0.02V単位, mV÷20)
        * `Byte 5`: **セル2電圧** (符号なし8bit整数, 0.02V単位, mV÷20)
        * `Byte 6`: **セル3電圧** (符号なし8bit整数, 0.02V単位, mV÷20)
        * `Byte 7`: **電源動作ステート** (`0` = STOP, `1` = STANDBY, `2` = DRIVE)

> **DLC 拡張についての注意:** 仕様 v1.0.0 では DLC 7 (Byte 7 なし) として実装に追随していたが、v2.0.0 で本来の設計意図である Byte 7 = 電源動作ステートを復活させる。PowerBoard / CU の両ファームを **同時更新** する必要があり、片側のみ DLC 8 化すると受信側の DataLength フィルタで弾かれて 0x258 が落ちる。リリース運用時は両ファームを揃えて反映すること。

### 2.5. パラメータ設定/読み取り要求

メイン基板から電源基板の保護閾値を設定/読み取りする。

* **メイン基板 → 電源基板：パラメータ設定/読み取り要求**
    * **CAN ID**: `0x280`
    * **DLC**: 4-8バイト

**パラメータID定義:**

| ID | パラメータ | 単位 | 範囲 | デフォルト値 |
|----|-----------|------|------|-------------|
| `0x00` | 過電流閾値 (OVERCURRENT_THRESHOLD) | mA | 0-65535 | 10000 |
| `0x01` | 過電流継続時間 (OVERCURRENT_DURATION) | ms | 0-65535 | 5000 |
| `0x02` | 過放電閾値 (OVERDISCHARGE_THRESHOLD) | mV | 0-65535 | 3300 |
| `0x03` | 低電流閾値 (LOW_CURRENT_THRESHOLD) | mA | 0-65535 | 1000 |
| `0x04` | 過電流保護オーバーライド (OVERCURRENT_OVERRIDE) | - | 0-1 | 0 (保護有効) |
| `0x05` | 過放電保護オーバーライド (OVERDISCHARGE_OVERRIDE) | - | 0-1 | 0 (保護有効) |

**オーバーライドフラグの値:**
- `0`: 保護機能**有効**（通常動作）
- `1`: 保護機能**無効**（オーバーライド中）

**パラメータ設定コマンド (CMD=0x01):**

| Byte | 内容 | 説明 |
|------|------|------|
| `0` | コマンド | `0x01`: パラメータ設定 |
| `1` | パラメータID | `0x00`-`0x05` |
| `2-3` | パラメータ値 | uint16, リトルエンディアン |
| `4-7` | 予約 | `0x00` |

**全パラメータ読み取り要求コマンド (CMD=0x02):**

| Byte | 内容 | 説明 |
|------|------|------|
| `0` | コマンド | `0x02`: 全パラメータ読み取り要求 |
| `1-7` | 予約 | `0x00` |

### 2.6. パラメータ応答

電源基板はパラメータ設定/読み取り要求 (`0x280`) を受信すると、以下の応答を 3 フレームで返送する。フレーム識別はペイロード `Byte 1` の「フレーム番号」で行う。

* **電源基板 → メイン基板：パラメータ応答**
    * **CAN ID**: `0x2D8`
    * **DLC**: 8 バイト（3フレーム）

**フレーム0: 過電流系パラメータ**

| Byte | 内容 | 説明 |
|------|------|------|
| `0` | 結果コード | `0x00`=成功, `0x01`=無効なパラメータ |
| `1` | フレーム番号 | `0x00` |
| `2-3` | 過電流閾値 | uint16, mA, リトルエンディアン |
| `4-5` | 過電流継続時間 | uint16, ms, リトルエンディアン |
| `6` | 過電流保護オーバーライド | 0: 保護有効, 1: 保護無効 |
| `7` | 予約 | `0x00` |

**フレーム1: 過放電系パラメータ**

| Byte | 内容 | 説明 |
|------|------|------|
| `0` | 結果コード | `0x00`=成功, `0x01`=無効なパラメータ |
| `1` | フレーム番号 | `0x01` |
| `2-3` | 過放電閾値 | uint16, mV, リトルエンディアン |
| `4-5` | 低電流閾値 | uint16, mA, リトルエンディアン |
| `6` | 過放電保護オーバーライド | 0: 保護有効, 1: 保護無効 |
| `7` | 予約 | `0x00` |

**フレーム2: MACアドレス**

| Byte | 内容 | 説明 |
|------|------|------|
| `0` | 結果コード | `0x00`=成功 |
| `1` | フレーム番号 | `0x02` |
| `2-7` | MACアドレス | 6バイト |

### 2.7. HIDから電源基板への直接コマンド

HIDが電源基板へ直接送るコマンド。緊急停止やロボットID設定など、メイン基板を介さずに電源基板を制御する経路として運用する。EMS を含むため §1 のメッセージ種別表で `00010` (`0x088`) を割当て、`0x200` 系より高優先度に置く。

* **HID → 電源基板：HID直接コマンド**
    * **CAN ID**: `0x088`
    * **DLC**: 1～2 バイト

**コマンド種別 (Byte 0):**

| 値 | 名称 | 用途 | 後続データ |
|------|------|------|-----------|
| `0x01` | ロボットID設定 | 電源基板にロボットIDを設定 | `Byte 1`: ロボットID (uint8) |
| `0x02` | ロボットID/状態取得 | 電源基板から現在のID/状態を取得 (応答要求) | なし |
| `0x03` | 緊急停止 | HID由来の非常停止を発動 (`0x258` Byte 0 bit1 が立つ) | なし |
| `0x04` | 緊急停止解除 | HID由来の非常停止フラグをクリア | なし |
| `0x05` | 保護オーバーライド設定 | 過電流/過放電保護オーバーライドフラグを設定 (§2.5 のパラメータ `OVERCURRENT_OVERRIDE` / `OVERDISCHARGE_OVERRIDE` を直接書き換える) | `Byte 1`: ビットフラグ (bit0 = 過電流保護オーバーライド, bit1 = 過放電保護オーバーライド; いずれも `0` = 保護有効, `1` = 保護無効) |

> **保護オーバーライド設定の経緯:** 仕様 v1.0.0 では HID から保護オーバーライドを設定する正規ルートが定義されておらず、HID 側の暫定実装が §2.5 の `0x201 PARAM_CMD_SET` を直接送ることで代替していた (送信元が「メイン基板」限定という仕様と乖離)。v2.0.0 でこの 0x05 サブコマンドを正規ルートとして定義し、同時に旧 ID 例外 (`0x210` / `0x218`) を排除して新 ID (`0x088` / `0x0D8`) へ統一する。HID 側の実装は PowerBoard fw が 0x05 と新 ID をサポートした段階で切り替えること。

### 2.8. 電源基板からHIDへの応答

電源基板はHIDからのHID直接コマンド (`0x088`) を受信すると、以下の応答を返送する。

* **電源基板 → HID：HID直接応答**
    * **CAN ID**: `0x0D8`
    * **DLC**: 4 バイト
    * **データ**:
        * `Byte 0`: **結果コード** (`0x00`=成功, `0x01`=失敗)
        * `Byte 1`: **ロボットID** (uint8)
        * `Byte 2`: **電源動作ステート** (0=STOP, 1=STANDBY, 2=DRIVE)
        * `Byte 3`: **停止理由** (8bit ビットフラグ。§2.4 と同一定義を参照)


### 2.9 メイン基板パラメータ設定

* **WiFi → メイン基板：メイン基板パラメータ設定**
    * **CAN ID**: `0x110`
    * **DLC**: 8 バイト
    * **データ**:
        * `Byte 0-3`: **パラメータID**(4byte, 文字列4文字, 印字可能ASCII文字（0x20–0x7E）のみ)
        * `Byte 4-7`: **パラメータ値**(4byte)

### 2.10 メイン基板パラメータ設定返答

* **メイン基板 → WiFi：メイン基板パラメータ設定返答**
    * **CAN ID**: `0x780`
    * **DLC**: 6 バイト
    * **データ**:
        * `Byte 0` :  **ヘッダー**(0x01固定, パラメータ設定返答を表す) 
        * `Byte 1-4`: **パラメータID**(4byte, 文字列4文字, 印字可能ASCII文字（0x20–0x7E）のみ)
        * `Byte 5`: **設定結果**(0: 設定完了, 1: 設定値が不正, 2: 未対応パラメータ)

### 2.11 メイン基板パラメータ読み出し要求

* **WiFi → メイン基板：メイン基板パラメータ読み出し要求**
    * **CAN ID**: `0x150`
    * **DLC**: 4 バイト
    * **データ**:
        * `Byte 0-3`: **パラメータID**(4byte, 文字列4文字, 印字可能ASCII文字（0x20–0x7E）のみ)

### 2.12 メイン基板パラメータ読み出し応答

* **メイン基板 → WiFi：メイン基板パラメータ読み出し返答**
    * **CAN ID**: `0x780`
    * **DLC**: 8 バイト
    * **データ**:
        * `Byte 0-3`: **パラメータID**(4byte, 文字列4文字, 印字可能ASCII文字（0x20–0x7E）のみ)
        * `Byte 4-7`: **パラメータ値**(4byte)

> **メイン基板パラメータ設定返答と読み出し応答の区別** ヘッダー(byte 0)が0x01かどうかで判定。読み出し応答における byte 0 はパラメータID先頭文字（印字可能 ASCII 0x20–0x7E）のため 0x01 にはならず、区別できる。また、DLCでも判別できるがあくまで補助的に利用。

## 3. データ型とエンディアン

* **エンディアン**: すべてのマルチバイトデータは**リトルエンディアン**で表現されます。
* **固定小数点数**: 電流、速度、角度、電圧などは、小数部を整数で表現するために適切なスケール（例: 電流はmA、電圧は0.01V、角度は0.01deg）を適用します。
* **符号付き/符号なし**: 各データの性質に応じて、符号付き整数か符号なし整数かを明確に定義しています。
